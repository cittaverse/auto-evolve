# 系统架构 (System Architecture)

## 整体设计

本研究采用**双引擎 + 记忆图谱**的分层架构（图 3-1），将对话引导与质量评估解耦，同时通过共享的状态表示保证两者的一致性。系统由三个核心组件构成：

1. **LLM 引导引擎（Conversation Engine）**：负责与用户进行多轮对话，基于当前叙事状态动态生成引导性问题
2. **评分引擎（Scoring Engine）**：负责对用户回复进行六维度质量评估，输出 L0 级细粒度分数
3. **记忆图谱（Memory Graph）**：作为共享状态存储，记录已提取的事件、实体、情感与时间线索

## 模块职责

### 3.1.1 Conversation Engine

引导引擎的核心任务是在**信息获取**与**认知负荷**之间取得平衡。过强的引导可能导致用户被动回答，丧失自传体回忆的自发性；过弱的引导则可能导致叙事碎片化，难以形成连贯的生命故事。

**Prompt Manager** 维护三类 prompt 模板：
- **开场类**：用于会话初始化，建立信任并说明任务目标（例："今天想和您一起回顾一下您的人生故事..."）
- **探索类**：用于深入特定主题，包含时间线索、感官提示、情感提示（例："那时候您多大？能描述一下那天的天气吗？"）
- **过渡类**：用于话题切换，保持叙事连贯性（例："刚才您提到了工作经历，那您的家庭生活呢？"）

**Context Builder** 负责构建每轮对话的上下文窗口，包含：
- 系统指令（system prompt）：定义 AI 角色、任务目标、边界约束
- 对话历史（conversation history）：最近 N 轮对话（默认 N=10）
- 记忆图谱摘要（graph summary）：已提取的关键事件与实体列表
- 当前策略（current strategy）：本阶段的引导重点（例："深入探索童年记忆"）

**Strategy Selector** 基于记忆图谱的状态动态选择引导策略：
- 若某时间段事件稀疏 → 触发"时间线补全"策略
- 若情感效价持续负面 → 触发"积极资源激活"策略
- 若感官细节不足 → 触发"五感提示"策略
- 若社会联结缺失 → 触发"关系网络探索"策略

### 3.1.2 Memory Graph

记忆图谱采用**属性图（Property Graph）**模型，节点表示事件或实体，边表示关系。设计原则是**轻量级、可增量更新、支持语义查询**。

**事件节点（Event Node）** 包含以下属性：
- `id`: 唯一标识符（UUID）
- `time`: 时间描述（模糊或精确，例："1985 年夏天" 或 "1985-06"）
- `location`: 地点描述
- `participants`: 参与者列表（指向实体节点的引用）
- `emotional_valence`: 情感效价（-1 到 1）
- `summary`: 事件摘要（50-100 字）
- `source_turn`: 来源对话轮次

**实体节点（Entity Node）** 包含：
- `id`: 唯一标识符
- `name`: 实体名称
- `type`: 实体类型（人物/地点/组织/物品）
- `description`: 实体描述
- `first_mention_turn`: 首次提及的对话轮次

**关系边（Relationship Edge）** 包含：
- `type`: 关系类型（参与/位于/属于/导致/之前/之后）
- `source`: 源节点 ID
- `target`: 目标节点 ID
- `confidence`: 置信度（0-1，由 LLM 评估）

记忆图谱的更新采用**增量抽取**策略：每轮对话后，LLM 从用户回复中提取新的事件与实体，与现有图谱进行实体对齐（entity resolution），避免重复节点。

### 3.1.3 Scoring Engine

评分引擎采用**三层评估体系**，从细粒度维度到整体质量逐层聚合：

**L0（维度级）**：六维度独立评分，每维度 0-5 分
- 时间连贯性（Temporal Coherence）
- 情感效价（Emotional Valence）
- 感官细节（Sensory Details）
- 社会联结（Social Connection）
- 事件分段（Event Segmentation）
- 自我反思（Self-Reflection）

**L1（会话级）**：对单 session 内所有 turn 的 L0 分数进行加权平均，输出该 session 的整体质量

**L2（用户级）**：对多 session 的 L1 分数进行聚合，考虑时间跨度、主题多样性等因素，输出用户叙事能力的综合评估

评分引擎与引导引擎共享同一 LLM 后端（Qwen-Plus），但使用不同的 prompt 模板。评分 prompt 采用**结构化输出**格式，要求模型输出 JSON 而非自然语言，便于程序解析。

## 数据流

系统的数据流遵循**请求 - 响应**模式，但引入了**异步评分**机制以优化用户体验：

1. **同步路径**（用户感知）：
   - 用户发送消息 → Conversation Engine 生成回复 → 返回给用户
   - 延迟要求：< 3 秒

2. **异步路径**（后台处理）：
   - 用户消息 → Scoring Engine 评估 → 更新记忆图谱 → 更新策略状态
   - 延迟要求：< 10 秒（用户无感知）

3. **持久化路径**：
   - 每轮对话结束后，将对话记录、评分结果、图谱快照写入数据库
   - 支持断点续聊与多设备同步

数据流的关键设计决策是**评分与引导解耦**：引导引擎不等待评分完成即可生成回复，确保响应速度；评分结果用于更新下一轮的引导策略，形成闭环反馈。
